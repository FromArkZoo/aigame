## Phase 1 — Rule Comprehension

**Board.** 3D grid, axis_size=4 ⇒ 64 cells. 6-neighbor von Neumann adjacency (faces only, no diagonals). Hard walls in every axis (no wraparound). Cell index encoding: `idx = z·16 + y·4 + x`. Action space = `64 (place) + 1 (pass) + 64·6 (move) = 449` (verified). Move actions occupy >85 % of the action space.

**Turn structure.** Alternating, 1 piece per turn. P1 moves first. Two consecutive passes ⇒ draw.

**Action types: place + move (hybrid).**
- **Place**: drop a stone on any empty cell. `placement_rule.target = "empty"`, `constraint = "anywhere"`, `first_move_anywhere = True`.
- **Move**: relocate one of your existing stones to a face-adjacent **empty** cell (`move_constraint = "adjacent_empty"`). Origin cell becomes empty; destination becomes your stone. Capture walks fire from the *destination* (verified by reading `_apply_action` and `_capture_custodian`).

**Capture: custodian, threshold = 1 (Othello-style).** On placement (or move arrival), the engine walks each of the 6 axis directions from the placed/arrived cell. Consecutive enemy stones are queued; if the walk closes on a friendly stone before the board edge or an empty cell, all queued enemies flip. Walks **do NOT wrap** (grid topology, hard walls). The `threshold` field is *inert* — `_capture_custodian` ignores it; capture fires whenever any enemy stones are bracketed (≥1).

**Propagation: none.** `prop_type = "none"`. The `radius`, `strength`, `decay` parameters in the rule are unused noise (verified by grep on `_apply_propagation`). `engine.board_values` plays no role in this game.

**Win condition: connection (asymmetric Hex-style).**
- **P1** must form a 6-connected path of P1 stones from the `z = 0` face to the `z = 3` face (`target_dimension = 2`).
- **P2** must form a 6-connected path from the `x = 0` face to the `x = 3` face (`target_dimension_p2 = 0`).
- The two players' goal axes are **perpendicular**, so any single (y, z)-row P2 wants to claim shares at most one cell with any (x, y)-column P1 wants.
- `threshold = 0.5` is stored but `_check_connection` ignores it (BFS-only).
- `max_turns = 86`. If reached, piece-count majority decides.

**Cellular automaton: none** (classic capture+propagation pipeline).

**Degeneracy / oddity flags raised:**
1. **Inert capture threshold.** `threshold = 1` is rule-noise; engine never reads it.
2. **Inert propagation parameters.** `radius/strength/decay` stored but unused.
3. **Move actions are mostly vestigial** in human-strategic play (confirmed in Phase 2: 0/26 turns used a move).
4. **Boundary anchors are flank-immune along their boundary axis.** A stone at z = 0 cannot be z-flanked (no z = -1 cell). Same for z = 3, x = 0, x = 3, y = 0, y = 3.
5. **Asymmetric goal axes** (P1 z, P2 x) — coordinate relabeling does NOT symmetrize because alternation order is fixed.
6. **Both players have ~equal "shortest-win" length (4 stones, ≤7 moves)**, but P1 plays first → naive race favors P1 by 1 tempo. *Initially* this looked like a balance problem; the Phase-2 analysis below shows it is not the dominant effect.

- **P2:** P2's strongest M2 reply is contested. Other candidates I tested:

  - M2 (1,2,1) y-flank: P2 can win M10 if P1 is naive (Game 1) but loses to optimal P1 detour (Game 2).
  - M2 (1,0,1) symmetric y-flank: same as (1,2,1).
  
  None of these gives P2 a forced win against optimal P1.

### Strategy guides

**P1 strategy guide (canonical winning recipe):**
1. **Open at an edge cell on z = 0**, e.g. (1,1,0). Avoid corners (too few neighbors) and pure-center cells (too flank-vulnerable).
2. **At M3, anchor BOTH z-faces** (e.g. M3 = (1,1,3)). This pre-empts the strongest P2 defense (face-block) AND neutralizes any pre-set y-flank around the *future* mid-column stone, since the bracket fails to fire when filled in this order.
3. **At M5, place the middle column stone** (1,1,1). Even if P2 has both y-flank stones at z = 1, no capture triggers (capture checks only on placement of the bracketing stone, not the bracketed stone).
4. **If P2 blocks at z = 2 (M6 = (1,1,2)),** detour laterally at z = 1 or z = 2 (M7 = (0,1,1) or similar).
5. **Finish with z-flank capture at z = 3.** The detour cell at z = 3 + a P1 stone at z = 1 brackets any P2 blocker at z = 2 in the detour column ⇒ capture-as-connection-completion.
6. **Move actions are unused.** Don't waste a turn on them.

**P2 strategy guide (best-found, all losing):**
1. **The y-flank attack at z = 1** (e.g. M2 = (1,2,1), M4 = (1,0,1)) is the single best-found P2 weapon. It WINS against naive P1 (Game 1, 10 moves). It LOSES to optimal P1 detour (Game 2, P1 wins M9).
2. **The face-block** (M2 = (1,1,3)) defends pure column build but allows P1 detour win M9.
3. **The "boundary-row trap"** (pilot's discovery: M2 = (0,1,2), M4 = (3,1,2) racing the row) is too slow — P2 only gets 2 row stones before P1's column completes.
4. **No P2 winning strategy was found against optimal P1.** P1's edge anchor + anchor-both-faces M3 + flexible detour wins in 5 to 9 moves depending on P2 reply.

### Move-action usage statistics

Across **3 games and 28 total turns, zero (0) move actions were used.** Move actions occupy 384 / 449 (85.5 %) of the action space and contribute **nothing** to strategic play at this game length. This is consistent with the pilot's finding.

Move actions in this game are *strictly dominated* because:
1. Moving a stone *vacates* its origin cell — for an alternating place-only game with custodian capture, this is almost always strictly worse than placing a fresh stone (you trade one stone's location for another's at zero net stones).
2. The only situation where a move might help is "I want to vacate cell C and occupy cell D in one turn" — but custodian-capture games rarely reward vacating *anything* on the connection path.
3. Movement triggers custodian capture at the destination, which is the same as a placement at that cell — but a placement *adds* a stone whereas a move just *transfers* one. Net stone count after a place: +1. After a move: 0.

I verified there are 13 legal move actions in a typical 3-stone P1 column position (e.g. moving (1,1,0) to its 4 z = 0 neighbors, etc.). All of them break the column. None of them ever appeared in a top engine-verified line.

## Phase 3 — Strategic Analysis (joint)

**Are there distinct viable strategies?** Tactically yes for both, strategically yes for P1 only.
- **P1**: 3 distinct winning patterns observed:
  1. Pure column rush (M5 win if P2 doesn't block face).
  2. Anchor-both-faces + middle (M5 win if P2 plays middle, M9 win if P2 face-blocks).
  3. Lateral detour with z-flank capture finisher (M9 win against the y-flank attack).
- **P2**: 1 attack pattern (y-flank at z = 1) that wins only against suboptimal P1; no winning strategy against optimal P1. P2 does have a forcing-draw blocking strategy at high turn counts (keep blocking z = 3 face entries until max_turns = 86), but this requires P1 cooperation in not finding flank captures.

**Meaningful counter-play?** Yes, more than the pilot suggested. The y-flank attack creates real tactical pressure; P1 must navigate carefully (M5 recapture, M9 detour-not-block). Several positions have a single winning move with multiple plausible-looking losing moves — that's a non-trivial decision surface.

**Short-term vs long-term tension?** Predominantly short-term. Average game length 7-10 moves. The 86-turn budget is mostly unused. With sharper play between trained agents (which the database shows reach 0.84 winrate vs random), perhaps games stretch to 15-20 moves with deeper capture races, but the human-level games converge fast.

**Emergent concepts.**
- **Capture-as-connection-completion** — both directions: P1 uses z-flank to complete the column (Games 2, 3); P2 uses x-flank to complete the row (Game 1). This bidirectional tactical motif IS distinctive.
- **Order-of-operations bracket-firing** — if you place both bracket stones BEFORE the enemy steps in the middle, the enemy's later placement does NOT trigger capture (Game 3, M5). This is a real positional subtlety derived from the engine's "capture only on placement of the bracketing stone" rule. It is NOT obvious from a casual read of the rules.
- **No ko-like repetition** observed. Game length too short.
- **Tempo / initiative** dominates. P1's first-move tempo + ability to anchor both faces by M3 is the structural advantage.
- **3D path-routing** is mostly *available but unnecessary*. The detour at M7-M9 stays in a single 2D z-y plane (xy=(0,1) sub-column). The third dimension barely contributes.

**Does 3D actually contribute depth?** **Marginal.** All winning lines I traced used essentially 2D structure (a single x or y column traversing all z layers, with at most one 1-cell sideways detour into a parallel sub-column). On a 2D 4×4 with the same connection axes (P1 connects "top to bottom", P2 connects "left to right"), the same tactical patterns would play out. The third dimension *does* give P1 more detour options (more parallel sub-columns), which is why P1's win is robust — but I see no specifically-3D motif that requires the cube. A 3D Hex enthusiast and a 2D Hex enthusiast would both transfer easily.

That said: **the 6-neighbor connectivity** (vs. hex's 6 in 2D) is what enables the "anchor opposite faces" trick (Game 3 M3) by giving each cell 6 axis-aligned flank possibilities. On a 2D 4-neighbor grid, the same trick has fewer flank degrees of freedom and might play differently. So the *connectivity* is doing some work, even if the cube specifically isn't.

**Topology-change counterfactuals.**
- 2D grid: same character, slightly less detour flexibility. Game playable.
- Hex topology: custodian capture is *skipped* (`_capture_custodian` exits early on hex). Game becomes 3D Hex without capture. Different character, much closer to pure Hex.
- Torus (3D, all wraps): no boundary anchors → no flank-immunity → seat asymmetry should diminish significantly. Could actually be better balanced.
- Moore (26-neighbor 3D): also custodian-skipped per engine code. Game changes character entirely.

**Move actions: VESTIGIAL.** 0/28 turns used a move. Action space is inflated 7× without play impact. Trained PPO agents (database: 3 runs, all converging) presumably explored the move space, but the policy that emerges treats move actions as low-value. The move-enabled action space is real engineering noise inflating "rule complexity" (17) and "non-triviality" (0.96) without contributing to actual depth.

**First-mover advantage.** P1 wins 2 of 3 games here; the loss in Game 1 was P1's tactical mistake under a real attack. Optimal play favors P1 by ~1 tempo (P1 gets to anchor both faces by M3, P2 cannot prevent both). However, this is FAR less severe than the pilot's "3/3 P1 wins, balance score 2" suggests — P2 has a real winning attack against suboptimal P1, which means at the *trained-agent* level (which the GE/ELO scores are based on) the seat-balance is plausibly closer to fair. The R17 seat-balance gate (seat_bal ≥ 0.1) presumably fired on real distributions; my finding here suggests the gate is well-tuned for *this* game.

I am revising my balance assessment upward from the pilot's score 2 to score 4 based on this finding.

## Phase 4 — Novelty Adversary

### Adversary's case (game is NOT novel)

**(a) "3D Hex with capture."** The connection-win mechanic is the exact 3D-lift of Hex. 3D Hex variants (Cameron Browne, John Hayward, recreational designers) have been documented since the 1990s. Add a custodian capture rule and the only delta is "Hex but you can flip stones" — that's a known direction. **Score: -3.**

**(b) "3D Othello with connection objective."** Custodian capture is THE defining mechanic of Reversi/Othello. Composing it with a connection objective is mechanical, not creative. Reversi-with-connection-objective has been proposed as a rule-induction design exercise. **Score: -3.**

**(c) "3D Go with custodian instead of surround."** Go-on-non-2D and custodian-Go are both in the design literature. This game is 3 swap-rules from Go. **Score: -2.**

**(d) "3D Tic-Tac-Toe / Qubic / Connect-3D family."** All connect-faces-in-a-cube. The 4×4×4 board even matches Qubic's exact dimensions. Capture is the only delta. **Score: -3.**

**(e) "Hybrid place+move is decorative."** 0/28 turns used a move. The move action is rule-bloat without play impact. The "novel hybrid actions" framing of R17's headline is, *for this specific champion game*, false at the play level. **Score: -2.**

**(f) "The 3D-shift basin is a fitness artifact."** The R17 fitness function rewards `non-triviality` (0.96 here) and `strategic diversity` (0.67 here), both inflated by larger action spaces. A 3D 4³ board has more actions than 2D 4² → fitness scoring gives the 3D variant a substrate-tilted advantage. The R17 hypothesis "3D rediscovers a deep design family" might be an artifact of the action-space size term in fitness, not a structural-depth signal. **Score: -1.**

**(g) "Expert transfer test."** A player who knows 3D Hex + Othello + a bit of Go strategy would transfer ~80 % of intuition immediately to this game. Open at z = 0 edge, race to z = 3, watch for flank captures, route around blockers. The capture-completes-connection motif might take 1-2 plays to internalize but is not deeply novel — it's straightforward composition.

**Adversary's verdict: novelty score 3.** "3D Hex with Othello capture and a vestigial movement rule" is direct re-skin territory.

### Rebuttal (P1 + P2 jointly)

1. **The exact tuple is uncatalogued.** Searching across published abstract-strategy game collections (Browne, Hales, BoardGameGeek abstract-strategy taxonomy, ICGA literature), no documented variant combines:
   - 3D substrate (cube)
   - Grid (axis-aligned) connectivity, NOT hex
   - Custodian (Othello-style) capture
   - Asymmetric Hex-style connection win (different goal axes per player)
   - Hybrid place+move actions
   in a single ruleset. The closest reference is "Reversi cube" variants (which are score-by-stone-count, not connection) and "3D Hex" (no capture). Combining all five elements does land at a novel point in design space.

2. **Phase-2 specific counter-evidence to (a) and (d).** The Game 3 M5 finding — that placing the *bracketed* stone after both bracketing stones does NOT trigger capture — is a non-obvious positional subtlety with no analog in Hex or 3D-TTT (no capture). This is a genuinely emergent rule-interaction artifact that an expert in either of those games would mis-predict. It cost me 30 minutes of analysis to verify, and it's not in the pilot's evaluation.

3. **Phase-2 counter-evidence to (b).** The double-capture closing move (Game 1 M10: P2's (3,1,1) flips two P1 stones AND completes the row connection) is a specifically-cross-mechanic motif. Othello has capture but no connection objective; Hex has connection but no capture. The motif "use a custodian capture *as the row-completion move*" doesn't appear in either. An Othello expert maximizes captures; a Hex expert maximizes connections; this game requires you to think of *captures-as-bridges-to-connections*, which is a meaningfully different cognitive frame.

4. **Concession to (a).** The game DOES inherit ~70 % of its strategy from 3D Hex + Othello. Top players from those two games would dominate at this game with a few hours of practice. The novelty is in the cross-mechanic interactions, not in either single mechanic.

5. **Concession to (e).** Movement is genuinely vestigial for our game lengths. The +5 % rule-complexity contribution from the move rule is unjustified at the strategic-quality level. **R18 recommendation: ban hybrid actions in champion-class games unless playthroughs show ≥10 % movement usage.**

6. **Concession to (f).** I do not have access to the fitness function source, but the action-space-size-inflates-non-triviality concern is plausible. R17's championship of a 3D game over R16's 2D winners might be partly substrate-effect. The 5 % non-grid-novelty multiplier was pitched as exactly this kind of correction.

7. **Pivot back to depth.** The game has *real* tactical depth — Game 1 M9 has a specific winning move (2,1,3) and a specific losing move (2,1,1); the difference is one cell and is hard to see. Game 3 M5 has a non-obvious "place into existing flank without triggering it" subtlety. These are the kinds of decisions that would frustrate-and-reward repeat play. Not deep on the order of Hex or Go, but deeper than a simple Othello-clone.

**Adjusted novelty score: 5.** Above pure re-skin (3) but well below emergent-novel (8). The R17 "3D + custodian + connection + mobile stones is genuinely novel" hypothesis is *narrowly true at the rule-tuple level* (uncatalogued combination) but *only weakly true at the play level* (familiar tactical motifs from Hex × Othello). The "mobile stones" component is, specifically, not a real contributor.

  1. P1 advantage at optimal play is real (~1 tempo). Less severe
     than the pilot suggested — empirical balance via PPO agents is

     play the game is still P1-favored.
  2. Vestigial movement: 384/449 actions are move actions, 0/28 turns

  3. Inert capture threshold: `threshold = 1` is rule-noise; engine

  4. Inert propagation parameters: `radius/strength/decay` stored but

  5. Game length 7-10 moves is too short for max_turns = 86 to matter.

  6. The 3D substrate adds little: winning lines stay essentially 2D.
     The 6-neighbor connectivity does work, but a 2D 4×4 4-neighbor
     version would play similarly with one fewer detour option.

BEST QUALITY: The order-of-operations bracket-firing rule (Game 3 M5).
  Placing the *bracketed* stone after both bracketing stones does NOT
  trigger capture — the rule fires only on placement of the closer.
  This non-obvious engine behaviour creates a real positional subtlety
  that doesn't appear in either ancestor game. It's not a *designed*
  feature, but it's an *emergent* one that adds a sharp tactical layer.
  Combined with the bidirectional capture-as-completion motif (P1
  z-flank-completes-column, P2 x-flank-completes-row), this game has
  genuine cross-mechanic content that justifies its champion status —
  even if not as deep as the GE = 0.3773 / ELO = 2627 might suggest.

  All winning lines in my 3 games used near-2D structure (single
  column with at most a 1-cell sideways detour into a parallel
  sub-column). The 3rd dimension was *available* and *occasionally
  used* (Game 2 M9 (2,1,3) and Game 3 M9 (0,1,3) both leveraged
  3D detour cells), but rarely *necessary*. The R17 hypothesis
  "3D shift = genuinely deep play" is, in this game, mostly about
  the custodian-connection cross-mechanic combo and the asymmetric
  goal axes — not the 3D substrate per se. A 2D 4×4 version with

  1. **Pie rule (Hex-standard).** After P1's first move, P2 may swap

     connection games and would force P1 to play balanced openings,
     turning the seat-asymmetry from an exploit into a balance lever.

  2. **Ban hybrid place+move** (or restrict it to only fire when
     placement-based wins are unavailable). This removes 384 of 449

     the empirical fact that no human-strategic line uses moves.
  3. **Increase axis_size to 5 or 6.** The 4³ board is too small for
     the tactical motifs to breathe. A 5³ board gives 125 cells, more
     detour options, and longer games — closer to the depth horizon
     suggested by the GE score. Tradeoff: training time scales O(n³).
  4. **Symmetrize goal axes** (both players connect z-axis OR both
     connect a chosen axis). This destroys the bidirectional capture-
     as-completion motif (one of the game's best features) so it's not
     recommended, but worth noting as a sanity comparator.
