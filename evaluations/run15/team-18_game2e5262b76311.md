## PHASE 1 — RULE COMPREHENSION

### Board & Topology
- 2-D, axis-size 8, `topology_type = torus`. Total cells = 64.
- Neighbors are the four Von Neumann neighbours with wraparound on both axes. (Torus identifies y=0 ↔ y=7 and x=0 ↔ x=7 as adjacent, but **not** as the same cell.)
- 65 actions: 64 placements (cell id = `y*8 + x`) plus action 64 = pass.

### Turn Structure — **SIMULTANEOUS** (`pieces_per_turn = 1`)
- Both players submit one action per "round".
- Collision rule: if both target the same non-pass cell → mutual annihilation, nothing is placed (the cell stays empty, or keeps its prior occupant). See `engine_v2.step_simultaneous` lines 211–233.
- After placements the CA loop fires; post-CA, win conditions are checked.
- Double-pass → draw (new R15 rule).

### Placement Rule
- Target: `empty`.
- Constraint: `adjacent_to_own` (Von Neumann neighbour of one of your own stones, torus wrap counts).
- `first_move_anywhere = True` — the very first move of each player (or any time they have zero stones again, due to CA death) is unrestricted.

### Capture / Propagation
- `capture_rule.capture_type = none`.
- `propagation_rule.prop_type = none`. `board_values` is identically zero for the whole game — the shown threshold 0.5 never applies because the win condition is `connection`, not `threshold`. The threshold field is vestigial.

### Cellular Automaton
- 75 entries in the table; **6 non-identity entries** from the perspective of the acting player (friendly=acting, enemy=opponent):
  - `0,1,2 → 1` — empty with 1 friendly + 2 enemy neighbours becomes friendly.
  - `0,2,1 → 1` — empty with 2 friendly + 1 enemy becomes friendly.
  - `1,0,2 → 0` — own stone dies with 2 enemy neighbours and no friendly support.
  - `1,1,3 → 0` — own stone dies with 1 friend + 3 enemies.
  - `2,2,0 → 0` — enemy stone is removed when surrounded by 2 friendly + 0 enemy.
  - `2,3,1 → 0` — enemy stone is removed with 3 friendly + 1 enemy.
- `steps_per_turn = 2`, and R15 runs both P1- and P2-perspective per tick, so **4 CA sweeps per round**.
- The table is generator-enforced symmetric under swap (verified by inspection — the 1-entry and 2-entry deaths mirror correctly).
- Effective dynamics are mostly quiescent: empirically, in 200 random-vs-random games to termination (avg length 8.5 rounds) I saw CA change > 2 cells in only a handful of rounds. Games end by connection before CA can build structure.

### Win Condition — `connection`
- P1 must connect `target_dimension = 1` → cells with y=0 and y=7 in a same-owner chain.
- P2 must connect `target_dimension_p2 = 0` → cells with x=0 and x=7 in a same-owner chain.
- Check order (`engine_v2._check_connection`): iterates `for player in (1, 2)` and returns on the first satisfied side. **P1 wins ties** — directly analogous to the R15 pilot-flagged `_check_threshold` order bias.
- `max_turns = 100`; game hitting the cap ends by piece-count majority.

### Degeneracy Flags (BEFORE any playthrough)
1. **Torus + connection + 2-move wrap = trivial win.** A player only needs cells on y=0 AND y=7 (or x=0 AND x=7) in a connected component. On a torus, cells at (x, 0) and (x, 7) are already neighbours via wraparound. So a 2-stone chain suffices: move 1 at (x, 0), move 2 at (x, 7). Since the second move is adjacent (via wrap) to the first, the `adjacent_to_own` constraint is satisfied.
2. **P1 wins the iteration-order tie-break.** If both players complete their 2-move wrap on the same round, P1 is returned first by `_check_connection`.
3. **Double-race is Nash-unstable for P2.** P2 must either race (and lose the tie) or block P1 (and lose their own race, unless the block cell also happens to set up P2's wrap, which only happens at two intersection cells, namely (0,0) and (0,7) type overlaps).

### Reflections

**Player 1 reflection.**

- What I'd do differently: nothing — the strategy is near-unconditional. The only loss happens when P2 guesses P1's column and places on (x, 7); then a pivot is forced. I could hedge by playing a column P2 is unlikely to block — but since P2's choice is also simultaneous, there is no Bayesian edge other than avoiding the corners if I suspect P2 will block corners.
- Opponent surprise: Game 2's block was a real threat. I under-estimated how efficient the block is (a single P2 stone at (x, 7) simultaneously blocks P1 and seeds P2's own wrap).
- Endgame type: all three games resolved by the stated win condition, in 2 rounds. **No double-pass, no max-turns.** But the games were degenerately short — none reached the CA or any emergent play.

**Player 2 reflection.**

- What I'd do differently: the only winning line for P2 against a racing P1 is to place at exactly (x_P1, 7) on round 1, where x_P1 is P1's chosen column. This is a 1/8 guess. Against a random P1, P2's own wrap strategy wins 92% of the time because random P1 rarely reaches its own wrap.

- Endgame: same as P1 — all three games reached the stated win condition in 2 rounds.

## PHASE 3 — STRATEGIC ANALYSIS (Player 1 and Player 2 joint)

- **Viable strategies?** One — the 2-move torus-wrap. There is no second strategy competitive with it, because any alternative takes more rounds while the opponent can finish in 2.
- **Counter-play?** Effectively none. The only P2 response that wins against an optimal P1 is a 1-in-8 blind guess of P1's wrap column; even then, P1 can pivot and it only delays, unless P2's block cell also completes P2's own wrap (which happens only when the block cell sits at x∈{0,7} on y=7, i.e. (0,7) or (7,7)).
- **Short- vs long-term tension?** None. The game is a one-round-of-depth guessing game with a deterministic winning line for P1.
- **Emergent concepts?** The CA is defined but never triggers in typical games (avg 8.5 moves in random play, 2 in optimal play; empty-cell-births require 1+2 / 2+1 mixed neighbourhoods which need ≥3 stones adjacent, rare this early). No emergent territory, influence, or ko. Mutual-annihilation is mechanically available but strategically irrelevant because skipping a round = losing the race.
- **Topology matters?** Critically and catastrophically — the torus is what turns what would have been a 7-move Hex-on-8x8-grid connection into a 2-move wrap win. On a non-torus grid with the same rules, the minimum connection would require 8 stones and the game would play quite differently.
- **First-mover advantage:** Not eliminated. Measured seat-swap evidence: in games 1 & 3 where both players race optimally, P1 wins the simultaneous tie. In all three phase-2 games, the winner was determined by (a) P1's iteration-order advantage or (b) P2 landing a correct block. The simultaneous mechanic fails to neutralise the turn-order bias because the win check iterates players in a fixed order. This is the exact R15 pilot-flagged issue (`_check_threshold` order) surfacing in `_check_connection`.

**Seat-swap acknowledgement:** I played both seats as a single agent. My P1 reasoning (race) is symmetric to my P2 reasoning (race); both were constructed from the same structural analysis, so there is no hidden depth I missed by seat identity. The asymmetry is baked into the engine, not the evaluator.

## PHASE 4 — NOVELTY ADVERSARY

### Adversarial case that this game is NOT novel

**(a) Broad catalog comparison.**
- *Hex.* The win condition is literally Hex: each player connects opposite faces of a board. Hex is normally played on a rhombus (or hexagon); this game is Hex on an 8×8 Von-Neumann grid wrapped as a torus. That single topology switch is the only distinguishing move.
- *Y, Havannah.* Connection games with the same "connect faces / corners" template. Same family.
- *Gomoku / Connect6 / Pente.* These are connection-by-line-count, a subclass of the same connection-game framework.
- *Go-like games with CA perturbation.* The place+CA-step loop is identical in shape to Inertia-Go or background-automaton Go variants (cf. Drössler & Rosenfeld's "Go with automata" variants). The specific CA here is a truncated Life-like totalistic rule.
- *Life-like CA literature.* With friendly counts only (ignoring enemy), the "1,0,2→0 / 1,1,3→0" death rules are reminiscent of Conway's "die if <2 or >3 neighbours" (B3/S23), but with an interference term for enemy population. The birth rules `0,1,2→1 / 0,2,1→1` pattern-match "HighLife-style" asymmetric births.
- *Simultaneous-move games.* Diplomacy order resolution, Blotto (pick columns/rows under information symmetry), Gungo (simultaneous Go). The mutual-annihilation-on-collision rule matches Gungo exactly. The one-shot round-1 column guess is pure 8×8 Blotto (each player picks a cell; wins/ties resolved by rule).

**(b) CA-specifically.**
The transition table has 6 non-identity entries (0.5% of the full 5³ = 125 Life-like-generalised state space, adjusted for asymmetric f/e). The death rule `1,0,2→0` is an "isolated stone surrounded" custodian — ubiquitous in Go. `2,2,0→0` is a 2-adjacency custodian capture — also Go-like. Taken together, this CA is a **Go-style custodian capture rule with a twist** (3-enemy death `1,1,3→0`) plus two birth rules that rarely fire in practice. It is **not** a new Life-like CA — it is a rule-encoding of capture dynamics that a hand-authored Go clone would write as capture logic.

**(c) Simultaneous.**
The round-1 Blotto guess + round-2 wrap is isomorphic to a 2-round matrix game where P1 picks column x ∈ {0..7} and P2 picks row y ∈ {0..7}, then each wraps. If we tabulate outcomes we get the mixed equilibrium of a constant-sum game with P1 value +0.77 (≈184/200) assuming P2 plays uniformly and blocking is tie-breaking only.

**(d) Re-skin argument.**
This game IS Hex, under the transformation:
  1. Replace the rhombus with a torus.
  2. Preserve "connect opposite faces" but allow wrap-adjacency to count.
  3. Add a cosmetic CA layer that doesn't fire in typical play.
The "2-move forced win" emerges because on a torus, opposite faces are already adjacent — so "Hex on a torus" is a degenerate variant that Hex-literature (see Schensted & Titus, "Mudcrack Y & Poly-Y") explicitly warns against because of exactly this collapse.

**(e) Expert transfer.**
A Hex expert would score 10/10 here in 5 minutes. There is no Hex-unspecific skill the game rewards. The CA expert would contribute nothing because the CA doesn't engage in optimal play.

### Rebuttal from Player 1 + Player 2

Concrete moments where a Hex/Go expert analogy would mislead:

- **Game 2, round 1: the block+seed double-use of (0,7).** In standard Hex, there is no analog to a single stone that simultaneously blocks the opponent's connection and starts yours — Hex edges don't wrap, so edge cells don't dual-serve. This game DOES have it (via torus wrap), so a Hex expert's edge-heuristic misvalues (0,7) specifically.
- **Iteration-order tie-break.** In Hex, simultaneity doesn't exist, so there's no analog. A Gungo player might expect mutual-annihilation on simultaneous connection, but here the engine quietly awards P1. No Hex/Gungo heuristic covers this.
- **CA presence as a threat, even dormant.** A Hex expert would play committal moves trusting stones to persist; in this game, a Hex-expert deep line (say, building up to a connection across the middle over 10+ moves) would expose stones to `1,0,2→0` and `1,1,3→0` deaths the moment the opponent throws two adjacent enemies. This is concrete in principle — but I did not observe it in Phase 2 because games were too short.

Honestly, the rebuttal is weak. Point 1 is a direct consequence of the torus and doesn't add strategic depth; it just changes which cells are valuable. Point 2 is an engine artefact, not a design feature. Point 3 is theoretical — the CA never fires in the games that actually resolve.

### Novelty score: **2/10**.
This is Hex with two degeneracies (torus collapse, P1 iteration-order win) and a dormant CA. A Hex expert wins immediately; there is no Hex-orthogonal mechanic actually affecting play.

## PHASE 5 — VERDICT

Team ID: `team-18`
Game ID: `2e5262b76311`
Rules Summary: Simultaneous-move connection game on an 8×8 torus; place adjacent to your own stones; four custodian-style CA sweeps per round; first to connect opposite faces (y-faces for P1, x-faces for P2) wins.
Topology: 8×8 Von Neumann torus
Turn Structure: **simultaneous**, mutual annihilation on cell collision

### SCORES (1-10)

- **Strategic Depth: 2** — There is effectively one strategic decision per game (round-1 column/row pick) and one follow-up (the forced wrap). Post-round-2 the game is decided. Training reached avg length 25-35 only because self-play randomised the column and sometimes missed the wrap.
- **Emergent Complexity: 2** — Defined CA is 6 non-identity entries; random-play games average 8.5 rounds, optimal games average 2 rounds, and the CA rules that do fire (deaths on enemy crowding) are structurally equivalent to Go custodian capture. Zero emergent concepts arose in Phase 2.
- **Balance: 2** — Seat-swap evidence: P1 wins 191/200 in wrap-vs-wrap, 199/200 in wrap-vs-random; P2 wins 184/200 in reverse only because P1 is random and fails to race. The iteration-order tie-break on `_check_connection` (engine_v2.py:741) is the same family of bug as the R15-flagged `_check_threshold` bias. Simultaneous play did NOT neutralise first-mover advantage.
- **Novelty (post-adversary): 2** — Strongest adversary argument: this is literally Hex on a torus, with a dormant Go-style capture CA. Rebuttal was specific (round-1 dual-use cells, iteration-order oddity, theoretical CA threat) but concedes the core structure is Hex. The torus wraparound is a published Hex pathology, not a novel design.
- **Replayability: 2** — Once both players learn the 2-move wrap and the iteration-order tie-break, P1 wins nearly every game. There are 8 opening columns for P1 and 8 rows for P2, giving 64 symmetric openings, but almost all collapse to the same result.
- **Overall "Would I play this again?": 2** — Only as a teaching example of why torus + connection + simultaneous + first-iteration-wins-ties is a degenerate combination.

### CLOSEST KNOWN-GAME ANALOG
**Hex on a torus, simultaneous-move variant.** Not identical to Hex because (a) it's simultaneous with mutual annihilation and (b) the CA layer exists (even if dormant) and (c) the connection check is P1-favoured on ties. But the core — build a chain between opposite faces of a polygonal board — is pure Hex. Gungo is a secondary analog for the simultaneous + annihilation angle.

### KILLER FLAWS
1. **2-move forced win via torus wrap.** (x, 0) + (x, 7) is an adjacent 2-cell chain that connects the two opposite y-faces. Same on the x-axis for P2.
2. **P1 wins all simultaneous ties** because `_check_connection` iterates `for player in (1, 2)` and returns on the first match (engine_v2.py:741).
3. **CA is effectively dead in practice.** The 6 non-identity entries require neighbourhood patterns that don't arise in 2-round games, which are 91%+ of optimally played games.
4. **Propagation and threshold fields are vestigial.** `prop_type = none` ⇒ `board_values = 0` everywhere ⇒ the 0.5 threshold is unreachable; `condition_type = connection` ignores it anyway, but a reader seeing the printout is confused.
5. **`first_move_anywhere`** combined with the CA death rules creates a pathological re-opening: if CA wipes all of a player's stones (possible but rare), they can re-place anywhere, which could be exploited to jump to the opposite face again.

### BEST QUALITY
The one interesting thing is the **round-1 Blotto mini-game**: P2's (0,7) is a dual-use cell — it blocks P1's column-0 wrap AND seeds P2's row-7 wrap. Noticing this kind of dual-use cell is genuinely Hex-orthogonal, and if the torus collapse were fixed (see improvement below) this dual-use structure could become strategically rich.

### IMPROVEMENT IDEAS
**Change the topology to `grid` (or keep torus but change `connects_faces` to require the chain to traverse strictly more than `axis_size / 2` distinct rows/columns).** This single fix destroys the 2-move wrap because (x, 0) and (x, 7) would no longer be adjacent on a grid, requiring 8 stones to connect, and it would give the CA and simultaneity a meaningful runway to affect play. The secondary fix is to **randomise the iteration order of `_check_connection`** (or have simultaneous double-connection resolve as a draw) so simultaneous ties don't silently favour P1.
