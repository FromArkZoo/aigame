## PHASE 1 — RULE COMPREHENSION

### Plain-English summary

Two players play on an 8×8 torus (edges wrap in both x and y). Each round both players simultaneously pick an action: place a stone on an empty cell, or pass. The first stone a player places can be anywhere; all subsequent stones must be on an empty cell orthogonally adjacent (torus neighbors, so x=0 ↔ x=7 and y=0 ↔ y=7) to one of their own stones. If both players target the same empty cell the stones mutually annihilate — neither is placed, both lose that placement.

After placements are resolved, a cellular automaton runs for `steps_per_turn=2` ticks, and each tick runs the rule from BOTH players' perspectives (P1 first, then P2 — this is the R15 bug-fix: R14 would only fire P1's perspective). The rule takes `(center_state, friendly_count, enemy_count)` and returns a new state. After the CA, the engine checks whether either player's cells form a connected set reaching both opposite faces along that player's target axis — if yes, that player wins; if both simultaneously meet the condition, the `for player in (1, 2)` iteration order awards the win to **P1**.

Win condition: connection. P1 wins if their stones connect y=0 row to y=7 row (wrapped BFS restricted to own cells); P2 wins if their stones connect x=0 column to x=7 column. `max_turns=100`; double-pass is a draw.

### CA rule-table analysis (actually firing rules)

Prompt said "75/75 non-identity transitions" — this is wrong by a lot. Only **6 cells of the 75-entry table are non-identity**:

| abstract-state | fc (friendly) | ec (enemy) | output | meaning |
|---|---|---|---|---|
| 0 (empty) | 1 | 2 | 1 | birth friendly when 1-own + 2-enemy neighbors |
| 0 (empty) | 2 | 1 | 1 | birth friendly when 2-own + 1-enemy neighbors |
| 1 (friendly) | 0 | 2 | 0 | die when surrounded by 2 enemies and no allies |
| 1 (friendly) | 1 | 3 | 0 | die when outnumbered 1:3 by enemies |
| 2 (enemy) | 2 | 0 | 0 | enemy dies with 2 own + 0 friendly neighbors |
| 2 (enemy) | 3 | 1 | 0 | enemy dies with 3 own + 1 friendly neighbors |

The last two entries ("enemy dies when…") look redundant with the first two once you realise the engine symmetrises by running per-player perspective, but the generator's `T(1,f,e)=swap(T(2,e,f))` symmetry rule is satisfied — I verified this programmatically. The R15 claim of a symmetric table holds.

**Practical CA behaviour:**
- Births are narrow: exactly one (1,2) or (2,1) empty-neighbor configuration, which requires 3 of the 4 neighbors to be stones and the mix to be unbalanced by 1. Rare early game.
- Deaths require 2+ enemies and ≤1 friend. Also uncommon in normal play (your own stones cluster).
- In 50 random vs random games the CA fired on average ~2 events per game — a soft sprinkle, not a driving force.

### Flags

- **Not 75/75 non-identity.** Only 6/75 transitions are non-identity (8%). CA is active in a narrow sense — `steps_per_turn=2` multiplies events — but the rule table itself is sparse.
- **Iteration-order P1 bias in CA births.** Because `step_simultaneous` runs `_run_ca_step(1)` then `_run_ca_step(2)` per tick, when both players' perspectives would birth the same cell simultaneously (very common, since the two birth rules are mirror images), **P1 always wins the birth**. I verified this: a configuration of {1 P1 + 2 P2 neighbors around an empty cell} births a P1 stone at that cell, not a P2 stone, even though from P2's perspective it's {2 friendly + 1 enemy} which would also birth friendly.
- **Iteration-order P1 bias in connection check.** Same as the threshold check the prompt warned about: `_check_connection` iterates `for player in (1, 2)` and returns on first win. If both players complete their connection on the same tick, **P1 wins**. I verified this in Game 1.
- **Torus-wrap breaks face-connection games.** With torus topology, y=0 and y=7 are neighbors via wrap. So `connects_faces(dim=1)` with any single stone at y=0 and any single stone at y=7 forming a connected (through wrap) pair wins — if they are in the same x-column they are directly adjacent. This means **both players can often win on round 2** (first move anywhere, second move adjacent — and "adjacent" includes wrap-partners in the opposite face).

## PHASE 2 — STRATEGIC PLAY

All games played via `sim_play_helper.py` with engine-verified moves. Same agent plays both seats, with reasoning separated per move. Seat-swap in Game 3.

### Game 1 — P1 goes for corner-to-corner instant win

**P1 round 1 reasoning.** Any face cell on y=0 or y=7 enables a 2-round torus-wrap win. Open at (0,0) = action 0.

**P2 round 1 reasoning (after P1 committed).** I don't know P1 played (0,0); pick my own edge cell to mirror the strategy. (0,4) = action 32 — on x=0 face, leaves me an x=7 winning target later.

Engine result: P1 at (0,0), P2 at (0,4), no collision.

**P1 round 2 reasoning.** (0,0)'s torus neighbors are (7,0), (1,0), (0,1), (0,7). Playing (0,7) = 56 puts stones at y=0 AND y=7 faces, connected via wrap — instant win.

**P2 round 2 reasoning.** (0,4) neighbors are (7,4), (1,4), (0,3), (0,5). (7,4) is x=7 face, adjacent via wrap to (0,4) at x=0 face — instant win. Play (7,4) = 39.

Engine result: both placements legal, no collision. **Engine declares P1 winner** (iteration-order bias on simultaneous connection completion).

**Game 1 end:** P1 wins round 2 by tied-connection-race on iteration order. 2 steps total.

### Game 2 — P1 picks non-corner face, P2 tries to compete

**P1 round 1.** (3,0) = 3 — column 3, y=0 face.
**P2 round 1.** (0,3) = 24 — row 3, x=0 face. (Trying an orthogonal axis to reduce collision chance.)

**P1 round 2.** (3,7) = 59 — wrap-adjacent to (3,0); wins if unblocked.
**P2 round 2 (defensive attempt).** Try to block at (3,7) = 59 to collide with P1. **Engine rejects: action 59 is ILLEGAL for P2** because 59 is not adjacent to P2's only stone (0,3). P2 cannot block far from its own group.

P2's only legal moves: {16, 25, 31, 32, 64}. None block (3,7). Best competitive move is (7,3) = 31 — P2's own wrap-win.

Engine result: P1 places (3,7), P2 places (7,3). Both complete connection simultaneously. **P1 wins by iteration order**, 2 steps.

**Key discovery:** Because `adjacent_to_own` is enforced after the first move, **P2 cannot block P1's round-2 winning cell unless P2's round-1 stone was already planted next to it**. Committing in the dark gives P1 a first-move structural advantage even though turns are simultaneous.

### Game 3 — Seat swap. P1 plays a suboptimal center; P2 plays to exploit

**P1 round 1 (suboptimal on purpose).** (3,3) = 27 — central, avoids faces.
**P2 round 1 (strong).** (0,0) = 0 — corner face cell, sets up a round-2 win.

**P1 round 2.** Extend from (3,3); play (3,2) = 19 (legal).
**P2 round 2.** (7,0) = 7 is torus-adjacent to (0,0) via wrap; x=0 face plus x=7 face → horizontal connection. Wins.

Engine: **P2 wins round 2**, 2 steps.

**Game 3 observation:** This game confirms the symmetry of the broken mechanic. Whoever plays face cells on rounds 1 AND 2 wins. It's not a P1-structural bias — it's a generic "ignore the center, rush face wrap" forced win.

### Additional probe — can P2 ever block by pre-emption?

**Probe:** P1=(0,0)=0, P2=(0,7)=56 on round 1 (P2 preemptively takes P1's winning cell).
**Result:** P1 cannot play (0,7) round 2 (occupied). But (0,7) is on x=0 face, and (7,7) is on x=7 face and torus-adjacent to (0,7). P2 plays (7,7)=63 and wins on round 2. **P2 wins in 2 steps.** The cell P2 "stole" doubles as P2's own 2-move-win setup.

**Conclusion of Phase 2:** the game is a forced 2-round win under any opening where either player plays a face cell. Because face cells are on opposite sides simultaneously (corners belong to both "opposite face" pairs), opening on a corner gives the opener only a tiebreak-order advantage, not a structural one. None of the three games reached step 3. No collisions occurred in my 3 games (collisions require risky mirroring). CA never fired in any of the 3 games.

Double-pass draws: 0 of 3.

**P2 strategy (1 sentence):** open on x=0 or x=7 face (ideally a corner, which also serves P1, so it doubles as denial) and on round 2 play the wrap-opposite x-cell — if P1 has already committed to a non-face or a suboptimal cell you win; otherwise you tie and lose by iteration order.

## PHASE 3 — STRATEGIC ANALYSIS (joint)

**Distinct viable strategies?** No. The only coherent play pattern is "play face cell round 1, play wrap-opposite face cell round 2". Centre play is a blunder. There's no second strategy that wins against optimal play.

**Meaningful counter-play?** Almost none. P2's only real option is (a) collide on round 1 to deny (re-starts both players), (b) preempt P1's projected winning cell (risks giving P2 their own winning setup for free, as in the extra probe — actually that *helps* P2 if the cell doubles). (c) race on simultaneous tick and lose the tiebreak.

**Short-term vs long-term tension?** None. Games are 2 steps. No sacrifices, no positional build-up.

**Emergent concepts?** Nothing emerges. The CA never fires — we never saw a birth or death in the three evaluation games. The mutual-annihilation collision mechanic never fired either (both players chose distinct winning cells).

**Does topology matter?** Catastrophically — the torus wrap is the primary bug. On a grid topology the same rule would require the standard Hex-like bridge construction taking many moves. On torus, opposite faces are directly adjacent, collapsing the game to a 2-move race.

**First-mover advantage.** Quantified: 3/3 evaluation games won in 2 steps; in 20 random-vs-random games with seed=42 P1 won 60%. In a larger 50-game random bucket (excluding pass) P1 won 54% — so the iteration-order bias is real but partly masked by random play that often wastes the 2-move win. Under optimal play P1 wins 100% by iteration priority when both race faces.

## PHASE 4 — NOVELTY ADVERSARY

### Adversary case: this game is not novel

**(a) Known-game catalog.** This is Hex played on an 8×8 torus with simultaneous moves and a cosmetic but near-dead CA. Hex: two players place stones on a grid, each aiming to connect their opposing faces via a connected path of own stones. That is *precisely* the win condition here. The `adjacent_to_own` constraint is not in Hex, but Hex has no such constraint only because Hex already requires the path to be connected at the end; the path grows naturally. Our "adjacent_to_own" is a minor search-space restriction, not a new primitive. **Closest match: Hex, with a torus board and simultaneous moves.**

**(b) CA literature.** This rule table is a hollow degeneracy. Only 6/75 transitions are non-identity. The birth condition (exactly 1 vs 2 neighbors of unbalanced mix) is reminiscent of Conway's Life's B3 (birth with exactly 3 live neighbors) but with player identity baked in. With `max_neighbors=4` on a 4-neighbor torus, the CA is closer to a 2-state Von Neumann rule than anything resembling Life. And crucially — because the rule table reacts only to the very specific (1,2)/(2,1) configurations, in our 3 evaluation games the CA NEVER FIRED. It is a cosmetic decoration on top of Hex-on-a-torus.

**(c) Simultaneous-game literature.** Simultaneous Hex ("Gungo-like" variants) and Blotto-style blind allocation both involve simultaneous commitment. The mutual-annihilation-on-collision rule is the standard sim-placement resolution used in Gungo and in MIT's early simultaneous-Go experiments. Not new.

**(d) Re-skin transformation.** Let `f: torus-8×8 → hex-distorted-Hex-boardXL` map every 4-neighbor cell to a 4-neighbor graph node; the resulting game is "Hex with 4-neighbor adjacency and torus wrap". That's a 5-minute modification to Hex, not a new abstract game.

**(e) Expert crossover.** A strong Hex player, told "it's Hex on a torus with 4-neighbor adjacency and simultaneous moves", would immediately spot the torus wrap collapsing the game to trivial 2-move races. Their Hex reflexes (bridge constructions, ladders, Y-template theory) would be useless here *not because the game is deep* but because the topology breaks Hex's depth. The expert gets an immediate advantage by noticing the degeneracy, not because Hex skills transfer.

### Rebuttal (Players 1 and 2)

Referencing specific Phase-2 moments:

1. Game 1, round 2: both players complete connection on the same tick and P1 wins by iteration order. In real Hex this configuration cannot occur (alternating moves). This *is* a genuinely sim-specific scenario — the iteration-order tiebreak is a novel pathology, though a broken one rather than a feature.
2. Game 2, round 2: P2 could not play cell 59 because of `adjacent_to_own`, meaning *defensive blocking is topologically restricted by own prior placements*. Hex has no such restriction. This is a new mechanic, though again its net effect is to weaken P2 (not create depth).
3. Game 3, extra probe: P2 taking (0,7) on round 1 both (a) denies P1's natural winning cell and (b) sets up P2's own win at (7,7). This "a cell that's a face for one player is also a face for the other" is a corner-double-threat mechanic novel to the 2D torus. Hex has no analog, because Hex faces don't share.

However, these "novelties" are *pathologies exposed by a broken game*, not generative strategic depth. The adversary is correct that the closest known game (Hex-on-torus) is only a cosmetic rule-change away.

### Jointly-awarded Novelty score: **3**

The game is essentially simultaneous Hex on a torus with `adjacent_to_own` and a dead CA. The sim mechanic is standard; the torus is a reskinning; `adjacent_to_own` is a minor variant (Connect6 has similar local constraints). The CA does not actually contribute — it fires rarely and is never decisive. The only things that are genuinely novel are the pathologies (iteration-order tiebreak, `adjacent_to_own`-restricted blocking, corner-double-threat). Those are bugs, not features. Novelty ≈ 3.

- **Emergent Complexity: 1** — The CA never fired in the 3 evaluation games. Even in 50 random-play games it averaged ~2 events per game. Collisions fired 0 times in evaluation, rarely in random play (7/~450 rounds). No emergent structure arose.
- **Balance: 2** — Random-vs-random P1 win rate 54–60% depending on seed. Under optimal play, P1 wins 100% of tied simultaneous connection races by `for player in (1, 2)` iteration order (`_check_connection` in engine_v2.py:741). The "R15 fixes asymmetry" claim is contradicted by a new iteration-order bias (same class of bug as the threshold-check bias the prompt warned about — and, less important, an analogous bias in CA births). Training winrate 0.50 because both nets learned to play optimally and both pick face cells; the engine tiebreak is what awards the win.
- **Novelty (post-adversary): 3** — Closest analog is Hex on a torus with simultaneous moves. The CA is cosmetic and inert; `adjacent_to_own` is a minor variant of Connect6-style local-placement rules; sim-collision is standard. The only genuinely new effects (iteration-order tiebreak, adjacency-restricted blocking) are pathologies.
- **Replayability: 2** — Once you know the 2-move face-wrap pattern there is no reason to replay. Training logs show both seeds converged at win rate 0.5 only because both policies learned the same closed-form opening.
- **Overall "Would I play this again?": 2**

### CLOSEST KNOWN-GAME ANALOG
**Hex on an 8×8 torus with simultaneous moves** — not identical only because of (a) `adjacent_to_own` placement rule (Hex has no such restriction), (b) torus topology (Hex is on a rhombus), (c) a nominally present but inert CA. Strip those and you are playing Hex with a corrupted topology.

### KILLER FLAWS

1. **Torus-wrap collapses the connection game to a 2-move race.** Opposite faces are literally adjacent cells via wrap, so two stones (same column, y=0 and y=7) suffice for P1 to win and analogously for P2. Every one of our 3 games ended on step 2.
2. **Iteration-order P1 bias on simultaneous connection.** `_check_connection` iterates `for player in (1, 2)` and returns on first hit; both players completing on the same tick means P1 wins. Directly observed in Game 1. Same class of bug as the threshold-check bias the prompt warned about — the R15 changes did not fix connection.
3. **Iteration-order P1 bias in CA births.** Verified by direct engine probe: an empty cell with {1 P1, 2 P2} neighbors births a P1 stone on the P1 perspective tick and the P2 perspective tick then sees a P1 stone, so P2 never gets the symmetric birth. This would only matter if the CA fired meaningfully — it barely does — but the bias is present.
4. **CA is effectively dead.** 6/75 transitions non-identity; in 3 real games zero CA events occurred. Prompt metadata is also wrong (claims 75/75, reality is 6/75).
5. **`adjacent_to_own` eliminates defensive blocking** in simultaneous mode. P2 cannot block a P1 winning cell that is not adjacent to P2's existing stones — witnessed in Game 2 where `action 59 ILLEGAL for P2` rejected the intuitive block. Defense is only possible if P2's round-1 stone happened to be wrap-adjacent to P1's round-2 target, which P2 cannot know.

### BEST QUALITY
The mutual-annihilation collision mechanic is genuinely interesting *in principle* — it lets a trailing player deny a critical cell by colliding, and in sim-placement games this creates a min-max structure absent from alternating play. It simply never gets to act here because the 2-move forced win ends the game before any collision matters.

### IMPROVEMENT IDEAS
Strip the torus topology — make it a standard grid (or hex) — and the connection game becomes real Hex with meaningful bridge and ladder theory. The second-best fix would be to forbid the first move on any face cell (mirror the `adjacent_to_own` restriction to round 1), forcing both players to build inward before racing to a face. Either change would let the sim-collision mechanic and the CA actually matter, which is the whole premise of this branch of the R15 tree.

### Verdict on the R15 sim×CA hypothesis (the key question)
This game was the surviving sim+CA top-tier entry and it does **not** vindicate the premise. The R14 engine bug is fixed in the sense that both players' CA perspectives now run, but the CA itself is too sparse to matter in real play, a new iteration-order bias has migrated into the connection check, and the torus topology interacts disastrously with connection wins to make the game a 2-move forced win. Strategically meaningful sim×CA play did NOT emerge here. This evaluator's strong read is that R15's sim×CA signal in metrics is an artifact, not strategic substance — consistent with the broader R15 finding that sim×CA did not break through post-fix.
